## 背景

- GitHub Copilot等のAIエージェントが職場に導入される。
- 生成コードの品質安定化のために、特定チームの標準ルールを盛り込んだ `copilot-instructions.md` などを、複数部門間でgit共有する方式が検討される。

## 問題意識（静的ファイル共有の限界）

- **共通化の弊害**
  - いろんな技術スタックや文化が入り混じる組織で、単一の「共通の憲法」を使い回してうまくいくのか？
  - 誰にでも当てはまる標準化によって、内容が薄まってしまうのではないか？
  - 薄まった一般的な原則ならCopilotは既に知っている = 貴重なコンテクストウィンドウの無駄遣いになる。
- **運用の辛さ**
  - 組織の環境やベストプラクティスが日々変化する中で、画一的な標準化はすぐに陳腐化してしまう。
  - 実際に複数のプロジェクト間でgit共有方式（サブモジュール等）を試してみたが、割とめんどくさい。
    - サブモジュールのinit忘れ
    - 複数ブランチからサブモジュールを更新した時のコンフリクトやマージの手間
    - 結果、「めんどくさいから更新すんのやめとこ」と形骸化する。

## 課題設定

- とはいえ、優秀なエンジニアや組織が蓄積したinstructions（知見）を参照・再利用できることには大きな価値がある。
- このジレンマを解消するためには、以下の要件を満たす仕組みが必要。
  - 組織の「標準ルール」を容易に取り込める
  - 各プロジェクトの技術スタックや性質に合わせて「柔軟にカスタム（取捨選択）」できる
  - 開発者が「簡単に導入・変更（追従）」できる体験（DX）

## 結論

- 静的な完成品の `copilot-instructions.md` を共有するのではなく、**標準ルールの部品（フラグメント）群から必要なものだけを選択・結合してinstructionsを生成する「CLIツール（テンプレートエンジン）」を作成・配布**することにした。
- さらに、ツール側に複雑な選択ロジックを持たせず、**将来的にAI Agent自身にこのツールを使わせる（Tool Use）**ことで、プロジェクトに最適な指示書を全自動生成させる設計思想とした。

## 今後の展望：Agentにツールを使わせる（メタ・アプローチ）

実は、このCLIツール自体には「このプロジェクトがReactだからフロントエンドのルールを適用しよう」といった複雑な判定ロジックは実装していない。単なるテンプレートエンジンにとどめている。

理由は、現代であれば**「プロジェクトの性質を分析し、最適なルールを選択する」というタスク自体を、AI Agentに委ねることができる**からだ。

- **CLIツールにした真の理由**
  - UIを持たないシンプルなCLIツールにしたことで、Agentが Tool Use (Function calling) として非常に実行しやすい。
- **Agentによる全自動カスタマイズ**
  - 例：Agentにディレクトリ構成や `package.json` を読み込ませ、「このプロジェクトの技術スタックを分析し、最適な標準ルールフラグメントを選択してCLIツールを実行せよ」と指示する。
  - これにより、環境に完全にフィットしたinstructionsが全自動で生成される世界観が作れる。
- **品質の安定化**
  - AIにゼロからinstructionsを書かせるのではなく、CLIツールという「固定された良質なソース（選択肢）」を提供することで、ハルシネーションを防ぎ、組織として品質を安定化させられる。
